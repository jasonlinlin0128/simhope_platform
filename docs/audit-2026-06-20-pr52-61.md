### 🔴 P2 — 匿名濫用面（互相關聯，建議同批處理）

#### 1. C1 — 匿名 `/api/request` 可注入 Discord `@everyone` / 釣魚連結 〔workflow-verified, 3/3〕 ✅ Batch A 已修

- **證據**：`app/api/request/route.js:72-110`（feature 類型**免登入**）把使用者可控的 `message`(≤1000) / `name`(≤50) 原文塞進 `notify(...)`（line 107-109）；`src/lib/notify.js:13-18` 發 Discord `{content: text}` **沒有 `allowed_mentions`**。
- **影響**：任何匿名訪客（站台公開可達）送一筆需求，內容含 `@everyone`/`@here`/`<@&roleId>` → 直接 ping 整個內部 Discord；可注入 markdown 釣魚連結。`/api/admin/announce-tool`、`weekly-digest` 同樣經 notify，發布公告也會把工具標題裡的 mention 帶出去。
- **修法**：`notify.js` payload 加 `allowed_mentions: { parse: [] }`（一行，**一次保護所有 caller**）。
- **工時/ROI**：S（~15 min）。**最高 ROI**：一行修補擋掉匿名可達的內部騷擾面。

#### 2. C3 — `/api/track` 不驗證 `toolId` → analytics aggregate doc 無上限長 key 〔workflow-verified, 3/3〕 ⬇️ 重評 P3 → **Batch B**

- **證據**：`app/api/track/route.js:55-75` 用 `[inc.viewToolKey]` / `[inc.helpfulToolKey]` 對 `analytics/toolViews`、`analytics/toolHelpful` 做 `FieldValue.increment(1)`；`src/lib/trackEvents.mjs:47` 只 `String(toolId).slice(0,200)`，**從不比對真實目錄**。這兩個 doc **沒有 TTL**（只有 `analytics_daily` 設 expireAt）。
- **影響（已下修）**：~~污染排名/badge~~ → 偽造 id 不會出現在 UI（`rankPopularTools`/`attachHelpfulCounts` 皆只對**真實 toolId** 取值）。且限流不可繞過（見 C5 更正）。故實際影響縮為「匿名 POST 任意 toolId → 緩慢長 key，加上刪除/改名工具的孤兒 key（吞掉原 C12），長期逼近 Firestore **單 doc 1 MB 上限**後 view/helpful 寫入凍結」。低機率、慢。
- **修法**：`/api/track` 在寫 per-tool key 前，用一份輕量 catalog id 集合（`getServerCatalog` 已 ISR 快取）驗證 toolId 是否存在（不存在就只記總數、不寫 per-tool key，**fail-open**：catalog 取失敗時不過濾）；可選：上線一次性 cleanup 孤兒 key。
- **工時/ROI**：M（~半天）。資料完整性地雷但已非緊急 → **Batch B**（需小設計決策：在熱路徑加目錄查驗的 perf 取捨）。

#### 3. C2 — 公開 👍 badge + 🔥 排名可被匿名 `/api/track` 灌水 〔workflow-verified, 3/3〕 → **Batch B（需 brainstorm）**

- **證據**：`app/api/track/route.js`（tool_helpful / tool_view 匿名、僅 IP 限流）+ `src/components/HelpfulButton.jsx:45-55`（去重只靠 localStorage，可清）+ `src/lib/track.js`（client 端 session 去重，繞過 API 直打即失效）。badge 門檻 `helpfulBadge.mjs:5` = `≥3`，顯示在首頁/hub/chatbot（`ToolCard.jsx:134`）。
- **影響**：任何人直打 `/api/track` 就能把某工具刷過 ≥3 顯出「👍 N」**公開信任標章**，或灌 tool_view 拉高首頁熱門排名。**Phase-3 當初接受「不防灌水」是因為當時 helpful 只是後台隱形訊號；#58 把它變成對全員公開的標章 → 前提已改變**。

#### 4. ~~C5 — `clientIp()` 取可偽造的 XFF 最左段~~ ❌ **已更正為 REJECTED**

### 🟡 P3 — 韌性 / 正確性（低風險、便宜修）

#### 5. C10 — `notify()` 無 fetch timeout → Discord 卡住會拖垮整個 serverless invocation 〔self-adjudicated〕 ✅ Batch A 已修

- **證據**：`src/lib/notify.js:13` 純 `fetch(url, …)` 無 AbortController（對比 `gemini.mjs:37-56` 有 timeout）。`src/lib/fetchWithTimeout.mjs` 已存在卻沒用在這。

#### 6. C4 — `announce-tool` 即使 `notify()` 失敗仍寫 `announcedAt` → 暫時性 Discord 中斷＝公告永久遺失 〔workflow-verified, 3/3〕 ✅ Batch A 已修

- **證據**：`app/api/admin/announce-tool/route.js:44-48` — `await notify(...)`（回傳 false 不檢查）後**無條件** `ref.set({announcedAt}, {merge:true})`。
- **影響**：Discord 暫時不可用時，那一刻的「🎉 新工具上線」即時公告丟失、不重試。部分被 weekly digest（依 createdAt）兜底，但即時通知沒了。
- **修法**：只有 `notify()` 回 true 才寫 announcedAt；或記 retry 標記。工時 S。

#### 7. G5 — weekly digest 訊息無上限 vs Discord 2000 字限制 → 整則靜默丟失 〔self-adjudicated〕 ✅ Batch A 已修

- **證據**：`src/lib/weeklyDigest.mjs:29-35` `buildDigestMessage` 把所有近 7 天工具 join，無長度上限。
- **影響**：批次發布的一週（如 import 一次上 7 張、未來大批匯入）可能超過 2000 字 → Discord 回 400 → `notify` false → **整週摘要靜默不發**。
- **修法**：限制列數（如前 15 筆 + 「…還有 N 個」）或截斷。工時 S。

#### 8. G3 — Service worker 在 `/offline` 未快取時 `respondWith(undefined)` → 離線導覽變硬錯誤 〔self-adjudicated〕

- **證據**：`public/sw.js:31` `fetch(req).catch(() => caches.match(OFFLINE_URL))`；若 install 階段沒快取成功，`caches.match` resolve `undefined` → `respondWith(undefined)` 拋錯。
- **影響**：邊緣情境（precache 失敗 + 離線）下，離線頁失效、瀏覽器顯示原生網路錯誤。低機率。
- **修法**：cache miss 時 fallback 一個合成 `Response`。工時 S。

#### 9. a11y 小批（C6 + C8 + C9 + C7）— 新 UI 缺即時播報 / 表格語意 〔self-adjudicated〕

- **C6** `ChatbotWidget.jsx:101-125`：找工具結果 / 錯誤無 `aria-live` → 螢幕報讀者不知已回應。

- **C7** `ChatbotWidget.jsx`：浮動 popover 無 Esc 關閉 / dialog 語意（屬非 modal popover，較低；Esc 關閉為合理增強）。

2. **#52–#61 的純函式品質很高**：findTool / sortTools / toolSignals / weeklyDigest / announcePublish / helpfulBadge 全部防禦良好（非陣列/NaN/負值/壞輸入都安全退化），TDD 覆蓋紮實。問題集中在**路由邊界與外部副作用**（Discord、aggregate doc、限流），不在核心邏輯。
